# Buddy 발표 대본 최종 적용본

기준: 본편 1~19번 중심 10분 발표. 20번 이후 Q&A Prep과 Archive는 발표 중 읽지 않고 질문 대응용으로 사용합니다.

## 전체 메시지

Buddy는 작업 중 끊기는 맥락을 현재 화면 안에서 다시 이어주는 AI 작업 보조 도구입니다.

## 1. Title

안녕하세요. 모닥톤 2026 첫 번째 발표를 맡은 Buddy 팀입니다.

저희는 오늘 팀 이름과 같은 Buddy라는 서비스를 소개드리겠습니다.

## 2. Agenda

오늘 발표는 문제 정의에서 시작해서, Buddy가 어떤 서비스인지 먼저 소개드리겠습니다.

그 다음 핵심 기능을 설명드리고, 시연으로 실제 경험을 보여드리겠습니다.

시연 이후에는 방금 보신 경험을 어떤 구현 흐름과 역할 분담으로 만들었는지, 그리고 AI 토큰을 어떻게 활용했는지 말씀드리겠습니다.

## 3. Problem

저희가 출발한 문제는, AI를 사용하는 과정의 깊이였습니다.

화면에서 궁금한 것이 생기면 스크린샷을 찍고, AI 서비스로 이동하고, 이미지를 올리고, 다시 프롬프트로 맥락을 설명해야 합니다.

이 과정에서 작업 흐름이 끊기고, 같은 맥락을 반복해서 설명하게 됩니다.

그래서 저희는 "보고 있는 화면에서 바로 이어서 일할 수 없을까?"라는 질문에서 시작했습니다.

## 4. Project Message

그래서 저희가 정의한 주제는 "일하는 우리의 삶을 이어지게 만드는 작은 실험"입니다.

여기서 이어지게 만든다는 말은 사용자가 작업하던 화면을 벗어나지 않고, 끊긴 맥락을 바로 다시 잡을 수 있게 한다는 뜻입니다.

텍스트 추출, 번역, 요약, 이미지 설명, 자유 질문을 위해 매번 다른 서비스로 이동하지 않고, 현재 화면 안에서 도움을 이어받는 경험을 만들고자 했습니다.

그리고 필요한 맥락은 팀원과 공유할 수 있게 해서, 개인 작업뿐 아니라 함께 일하는 과정까지 이어지게 만들고 싶었습니다.

## 5. Solution Concept

Buddy는 화면 위에 상주하는 AI 작업 보조 도구입니다.

평소에는 화면 한쪽에 머무르다가, 사용자가 필요한 순간 화면을 캡처하면 내용을 이해하고 답합니다.

한 줄로 말하면, 캡처하고, 이해하고, 현재 화면 안에서 작업 흐름을 이어주는 친구입니다.

## 6. Feature Capture

첫 번째 핵심 기능은 캡처 기반 AI입니다.

사용자가 화면 전체나 영역을 선택하면, 그 화면이 AI의 입력으로 바뀝니다.

이를 통해 텍스트 추출, 번역, 요약, 이미지 설명, 자유 질문을 현재 화면 안에서 이어갈 수 있습니다.

또 캡처한 이미지 위에 펜이나 텍스트 주석을 남기고, 필요한 형태로 저장하거나 공유할 수 있게 했습니다.

목표는 반복되는 "캡처, 이동, 설명" 과정을 줄이는 것입니다.

## 7. Feature Persona

두 번째 핵심 기능은 페르소나 버디입니다.

같은 답변이라도 어떤 사용자는 핵심을 빠르게 정리해주길 원하고, 어떤 사용자는 맥락을 부드럽게 설명해주길 원합니다.

그래서 MBTI 기반으로 말투와 우선순위를 다르게 가져가도록 설계했습니다.

다만 페르소나는 사실을 바꾸는 것이 아니라, 답변의 톤과 표현 방식을 조정하는 역할입니다.

## 8. Feature Multi

확장 기능으로는 멀티와 클립보드를 검토했습니다.

Buddy는 개인 도구로 시작하지만, 팀원이 복사한 자료를 공유하거나 현재 상태를 이모지로 표현하거나, 서로의 Buddy를 화면에 같이 표시하는 방향으로 확장할 수 있습니다. 

즉 현재 화면 안의 도움을 넘어, 함께 일하는 맥락까지 이어주는 팀 도구로 확장할 수 있습니다.

## 9. Demo

이제 시연 영상을 보겠습니다.

데모에서 봐주실 포인트는 기능 개수보다, "AI를 쓰기 위해 작업 화면 밖으로 나가지 않아도 된다"는 경험입니다.

나의 MBTI를 선택하거나 간단한 MBTI를 검사하면 나에게 어울리는 MBTI를 추천받을 수 있습니다.

화면 캡처 기능을 사용하여 이미지 분석 버튼을 누르면 나에게 어울리는 MBTI가 사진 속 정보를 정리하여 알려줍니다. 글과 사진이 함께 있는 화면이나 사진만 있는 화면에 모두 활용할 수 있고, 원하는 언어로의 번역까지 요청함으로써 정보 검색에 대한 도구를 빠르게 사용할 수 있습니다.

이외에도 복사, 붙여넣기, 메모, 공유, 저장을 빠르게 할 수 있도록 멀티 클립 보드를 사용할 수 있습니다.

이렇게 저희가 제공하는 기능들을 사용하는 것으로 캐릭터의 레벨을 해금하여 다양한 커스터마이징 옵션들이 해금이 되구요, 다른 유저들이 있는 방에 입장하면 유저들이 취향껏 꾸민 캐릭터들을 볼 수 있고 이모지로도 소통할 수 있습니다.

## 10. Demo Video

여기서는 실제 영상을 보면서 Buddy가 어떤 순서로 동작하는지 확인하겠습니다.

핵심은 사용자가 화면 밖으로 나가지 않고, 캡처한 맥락 안에서 필요한 도움을 이어받는 흐름입니다.

영상이 끝난 뒤에는 이 경험을 어떤 구조와 역할 분담으로 만들었는지 설명드리겠습니다.

## 11. Implementation Flow

시연 화면만 보면 기능이 여러 개 나열된 것처럼 보일 수 있어서, 저희는 먼저 구현 순서를 세 단계로 정리했습니다.

첫 번째는 결정입니다. 화면을 벗어나면서 생기는 맥락 끊김을 줄이는 것을 MVP의 핵심 기준으로 잡았습니다.

두 번째는 구현입니다. 사용자가 직접 보고 조작하는 창, 캡처, 주석, 결과 확인 UI, 편집 UI는 Unity에서 만들고, 모델 호출과 결과 저장처럼 뒤에서 안정적으로 처리해야 하는 부분은 FastAPI 서버로 분리했습니다.

세 번째는 검증입니다. 모델 속도와 품질 비교, MBTI 말투 평가, UI 표준화 작업으로 시연에서 흔들릴 수 있는 부분을 보강했습니다.

즉 이 장표는 누가 무엇을 했는지보다, Buddy를 만들 때 어떤 순서로 문제를 좁히고 실제 동작까지 가져갔는지를 보여주는 장표입니다.

## 12. Team Role Breakdown

그 다음에는 이 구현 순서를 실제 작업 단위로 바꿨습니다.

사람을 먼저 나누기보다, 사용자가 화면을 캡처하고, 필요한 액션을 선택하고, 결과를 현재 화면에서 확인하는 유저 플로우를 기준으로 작업을 쪼갰습니다.

저희 팀은 프론트엔드, 게임 클라이언트, 서버, AI 활용 경험이 섞여 있었고, 각자 해보고 싶은 파트와 강점을 맞춰 역할을 맡았습니다.

그래서 Unity 클라이언트는 화면 위 경험을 만들고, 서버는 API와 모델 호출을 담당하고, 모델 테스트와 MBTI 검증은 답변 품질을 맞추고, UI 표준화는 여러 기능이 하나의 제품처럼 보이도록 정리했습니다.

이렇게 보면 각자 따로 만든 기능이 아니라, 하나의 유저 플로우를 완성하기 위해 장표처럼 나눠서 연결한 작업이라고 볼 수 있습니다.

## 13. Unity Client Contribution

Unity 작업은 해민, 승현, 영민 세 분이 나눠 진행했습니다.

해민은 프로젝트 세팅, 데스크톱 Buddy, 캡처와 주석, 무료 OCR/번역, 서버 연동을 맡았습니다.

승현은 Photon 룸 접속, 원격 외형 동기화, 이모지 RPC, macOS 캡처와 클립보드 포팅을 맡았습니다.

영민은 미션, 레벨, 파츠 해금, 치트 패널, Runtime UI 표준화로 반복 사용과 커스터마이즈 흐름을 정리했습니다.

세 작업을 합치면 Unity는 단순 화면이 아니라 Buddy의 실제 제품 경험 전체를 담당하게 됩니다.

## 14. AI Token Usage

이번 프로젝트는 AI 토큰 지원을 받은 만큼, 단순 코드 생성이 아니라 검증과 제작 속도를 높이는 데 집중적으로 사용했습니다.

현우는 모델 테스트, 정호는 MBTI 말투 검증, 영민은 Runtime UI 표준화, 해민과 승현은 Unity 개발과 Photon 연동 보조에 활용했습니다.

즉 AI를 대신 만들어주는 도구가 아니라, 팀의 반복 작업을 줄여주는 협업 도구로 썼습니다.

## 15. Model Evaluation

현우는 모델별 속도, 비용, 정확도와 MBTI 말투 재현을 비교해 기본 응답 전략을 정리했습니다.

핵심은 어떤 모델이 제일 유명한가가 아니라, Buddy의 실제 사용 흐름에서 어떤 모델이 빠르고 안정적인가였습니다.

그 결과 빠른 기본 응답은 GPT-5.4 mini 중심으로, 감정형 말투는 Claude Haiku를 참고하는 방향으로 정리했습니다.

## 16. PersonaForge Judge

정호는 MBTI 페르소나가 실제로 그 성격처럼 말하는지 검증하는 과정을 맡았습니다.

먼저 각 MBTI에 맞는 답변을 만들고, 답변에서 성격 이름표를 지운 뒤, 다른 AI에게 이 답변이 어떤 성격처럼 보이는지 맞혀보게 했습니다.

사람이 하나씩 읽고 판단하기 어려운 부분을 AI 심판으로 반복 확인한 것입니다.

이 과정을 통해 MBTI 말투가 단순히 라벨만 붙은 것이 아니라, 실제 답변 톤에서도 드러나도록 품질을 개선했습니다.

## 17. Runtime UI Standardization

영민은 여러 기능 화면의 UI가 제각각 보이지 않도록 표준화하는 작업을 맡았습니다.

Buddy는 캡처, 분석, 미션, 커스터마이즈처럼 화면 종류가 많다 보니 Canvas마다 버튼 위치, 색상, 닫기 방식이 조금씩 달라질 수 있었습니다.

그래서 영민은 AI에게 바로 코드를 맡기기보다, 먼저 공통 UI 명세를 만들고, 스크린샷을 보며 어느 위치와 규칙이 어긋났는지 하나씩 점검했습니다.

그 결과 여러 Canvas가 따로 만든 화면처럼 보이지 않고, 하나의 Buddy 제품 안에 들어온 화면처럼 보이도록 UI 톤을 맞췄습니다.

## 18. Roadmap

발표 이후에는 MVP를 제품 경험으로 확장하고 싶습니다.

지금은 Buddy에서 이어주는 플로우를 안정적으로 보여주는 것이 목표입니다.

다음 단계에서는 스크롤 캡처, 클립보드 히스토리, 최근 기능 모아보기, 단축키 지원처럼 사용성을 강화할 수 있습니다.

이후에는 프로젝트별 공유 히스토리, 팀 초대와 권한 관리, 작업 맥락 자동 요약처럼 팀 단위로 쓸 때 필요한 운영 기능까지 확장할 수 있습니다.
