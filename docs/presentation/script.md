# Buddy 발표 대본

> 최신 19장 본편 적용본은 `script-final.md`를 기준으로 사용합니다. 이 파일은 이전 대본과 Q&A/압축 제안 기록을 보존한 참고용입니다.

기준: 10분 발표. 현재 발표 자료는 47장입니다. 본편은 1~29번까지 빠르게 진행하고, 30번 이후 Q&A Prep 장표는 질문 대응용 백업으로 사용합니다.

## 전체 흐름 한 줄

Buddy는 "화면을 캡처하면 AI가 바로 이해하고, 현재 화면의 정보 처리와 동료와 함께 일하는 흐름까지 이어주는 AI 작업 보조 도구"입니다.

## 1. Title

안녕하세요. 저희는 모닥톤 2026 첫 번째 발표를 하게 된 팀 Buddy입니다.

저희는 오늘 팀 이름과 동일한 Buddy라는 서비스를 가져오게 되었습니다.

## 2. Agenda

10분이라는 시간 동안 저희가 어떤 문제에서 출발했는지, 그리고 Buddy가 어떤 프로젝트인지 소개드리려고 합니다.

오늘의 프로젝트를 어떻게 정의하게 되었고, 함께 모인 5명이 어떤 과정을 통해 프로젝트를 구현하고 만들었는지 다음과 같은 목차로 설명드리겠습니다.

## 3. Problem

저희가 출발한 문제는 AI 자체의 성능이 아니라, AI를 사용하는 과정의 깊이였습니다.

화면에서 궁금한 것이 생기면 스크린샷을 찍고, AI 서비스로 이동하고, 이미지를 올리고, 다시 프롬프트로 맥락을 설명해야 합니다.

이 과정에서 작업 흐름이 끊기고, 같은 맥락을 반복해서 설명하게 됩니다.

그래서 저희는 "보고 있는 화면에서 바로 AI 도움을 받을 수 없을까?"라는 질문에서 시작했습니다.

## 4. Project Message

그래서 정의내린 주제는 "일하는 우리의 삶을 이어지게 만드는 작은 실험"입니다.

여기서 이어진다는 말은 앱과 앱 사이, 화면과 AI 사이, 그리고 사용자와 도구 사이의 흐름이 끊기지 않는다는 뜻입니다.

조금 더 나아가, 텍스트 추출, 번역, 요약, 이미지 설명, 자유 질문을 현재 화면 안에서 이어주는 것뿐만 아니라, 동료와 함께 일하는 과정까지 이어주고자 했습니다.

저희는 AI를 더 멀리 있는 서비스가 아니라, 지금 작업 중인 화면과 함께 일하는 사람들 사이로 가져오고 싶었습니다.

## 5. Buddy Introduction

Buddy는 같은 곳을 바라보며, 마음까지 맞춰, 맥락과 질문을 이어주는 동료이고자 합니다.

현재 화면을 함께 보고, 사용자의 성향에 맞춰 필요한 답을 제시해주며, 필요한 맥락은 팀원과 공유되는 작업 흐름으로 확장될 수 있습니다.

## 6. Team Photo

이러한 프로젝트 방향을 구현하기 위해 Buddy 팀이 모였습니다.

저희는 프론트엔드, Unity 클라이언트, 서버처럼 서로 다른 직군의 팀원들로 구성됐습니다.

그래서 짧은 시간 안에 기획, 구현, 검증, 발표 준비를 병렬로 나눠서 진행할 수 있었습니다.

## 7. Team Members

그래서 역할은 각자의 강점에 맞춰 나눴습니다.

현우는 문서 관리와 발표 구조, AI 모델 테스트와 MBTI 데이터 수집을 맡았습니다.

해민, 승현, 영민은 Unity 클라이언트, 에셋 적용, UI 개선, Photon 기반 멀티 연동을 중심으로 작업했습니다.

정호는 MBTI 서버와 API 테스트, 클립보드 기능, AI 서버 구조를 맡았습니다.

## 8. Team Role Breakdown

이 장표는 각자의 작업이 어떻게 연결됐는지를 보여줍니다.

중요했던 점은 단순히 기능을 나눠서 만드는 것이 아니라, 캡처 기반 AI라는 핵심 플로우를 중심에 두고 클라이언트, 서버, AI 검증, UI 작업을 맞춰갔다는 점입니다.

## 9. Design Direction

해커톤 안에서 모든 것을 만들 수는 없기 때문에 목표를 세 가지로 좁혔습니다.

첫째, 핵심 기능은 캡처한 화면을 AI가 이해하고 바로 응답하는 흐름으로 잡았습니다.

둘째, 보는 즉시 제품 콘셉트가 이해되도록 비주얼 완성도를 확보하고자 했습니다.

셋째, MBTI 페르소나와 커스터마이징을 통해 단순한 도구보다 동료처럼 느껴지도록 했습니다.

## 10. Solution Concept

Buddy는 화면 위에 상주하는 AI 작업 보조 도구입니다.

평소에는 화면 한쪽에 머무르다가, 사용자가 필요한 순간 화면을 캡처하면 AI가 내용을 이해하고 말풍선으로 답합니다.

한 줄로 말하면 캡처하고, 이해하고, 질문에 답변해주는 친구입니다.

## 11. User Flow

사용 흐름은 먼저 프로그램을 켜고 닉네임과 서버 연결을 확인합니다. 그 다음 MBTI를 직접 고르거나 퀴즈를 통해 페르소나를 정합니다.

이후 Buddy가 화면에 등장하고, 사용자가 전체 화면이나 특정 영역을 캡처하면 번역, 설명, 질문 같은 AI 액션을 선택할 수 있습니다.

결과는 현재 화면 안에서 말풍선이나 결과 창으로 바로 확인합니다.

## 12. Feature Capture

첫 번째 핵심 기능은 캡처 기반 AI입니다.

사용자가 화면 전체나 영역을 선택하면, 그 화면이 AI의 입력으로 바뀝니다.

이를 통해 텍스트 추출, 번역, 요약, 이미지 설명, 자유 질문을 현재 화면 안에서 이어갈 수 있습니다.

목표는 반복되는 "캡처, 이동, 설명" 과정을 줄이는 것입니다.

## 13. Feature Persona

두 번째 핵심 기능은 페르소나 버디입니다.

같은 답변이라도 어떤 사용자는 핵심을 빠르게 정리해주길 원하고, 어떤 사용자는 맥락을 부드럽게 설명해주길 원합니다.

그래서 MBTI 기반으로 빠른 정리형, 맥락 공감형, 탐구 질문형처럼 말투와 우선순위를 다르게 가져가도록 설계했습니다.

다만 페르소나는 사실을 바꾸는 것이 아니라, 답변의 톤과 표현 방식을 조정하는 역할입니다.

## 14. Feature Multi

확장 기능으로는 멀티와 클립보드를 검토했습니다.

개인 도구로 시작하지만, 팀원이 복사한 자료를 공유하거나 현재 상태를 이모지로 표현하거나, 서로의 Buddy를 화면에 같이 표시하면서 함께 일하는 맥락도 이어주는 방향입니다.

이번 MVP에서는 핵심 플로우를 우선했고, 멀티 기능은 팀 도구로 확장할 수 있는 가능성으로 열어뒀습니다.

## 15. Demo

이제 시연 영상을 보겠습니다.

먼저 버디를 처음 쓰시는 분들은 MBTI 페르소나를 선택합니다.

그 다음 화면 일부를 캡처하고, 번역이나 설명 같은 액션을 실행합니다.

마지막으로 AI 응답이 현재 화면 안에서 결과 창이나 말풍선으로 이어지는 흐름을 보시면 됩니다.

데모에서 봐주실 포인트는 기능 개수보다, "AI를 쓰기 위해 화면 밖으로 나가지 않아도 된다"는 경험입니다.

## 16. Implementation Flow

개발 과정은 결정, 구현, 검증의 연속이었습니다.

앱 이동과 맥락 재설명이라는 병목을 MVP의 핵심 문제로 결정했습니다.

그 다음 Unity는 화면 위 경험과 편집 UI를 맡고, FastAPI 서버는 AI Key, 모델 호출, 결과 저장을 맡도록 역할을 분리했습니다.

마지막으로 모델 속도와 품질 비교, MBTI 말투 평가, Runtime UI 표준화로 서비스 안정성을 높였습니다.

## 17. Server Architecture

전체 구조는 Unity 데스크톱 클라이언트와 Mac mini FastAPI 서버로 나눴습니다.

Unity는 화면 캡처, 캐릭터, UI, 사용자 입력을 담당하고, 서버는 AI 모델 호출, persona 적용, 결과 저장을 담당합니다.

## 18. Technical Approach

기술 접근은 짧은 시간 안에 시연 가능한 조합을 기준으로 잡았습니다.

클라이언트는 Unity와 C#을 중심으로 데스크톱 앱, 캐릭터, 캡처, 에셋 활용을 검토했습니다.

AI 서버는 같은 Wi-Fi의 Mac mini에서 FastAPI로 실행했고, 모델 라우팅과 결과 저장을 맡겼습니다.

Persona는 MBTI 질문셋과 말투 프롬프트로 구성했고, Multi는 Photon으로 접속과 동기화를 검토했습니다.

## 19. Unity Client Experience

Buddy는 Unity를 통해 실현하고자 했습니다.

투명 오버레이 창 위에 2D 캐릭터가 떠 있고, 필요한 순간 호버 메뉴로 기능을 열 수 있습니다.

캡처, OCR/번역, AI 분석 결과, 말풍선 표시까지 Unity 안에서 이어지도록 구성했습니다.

또한 커스터마이징, MBTI 말투, 이모지 표현, Photon 기반 멀티 표현도 Unity 클라이언트가 담당했습니다.

## 20. Unity Client Contribution

해민, 승현, 영민 세 분이 Unity 작업을 나눠서 진행해주셨습니다.

해민은 프로젝트 세팅, 데스크톱 Buddy, 캡처와 주석, 무료 OCR/번역, 서버 연동을 맡았습니다.

승현은 Photon 룸 접속, 원격 외형 동기화, 이모지 RPC, macOS 캡처와 클립보드 포팅을 맡았습니다.

영민은 미션, 레벨, 파츠 해금, 치트 패널, Runtime UI 표준화로 반복 사용과 커스터마이즈 흐름을 정리했습니다.

세 작업을 합치면 Unity는 단순 화면이 아니라 Buddy의 실제 제품 경험 전체를 담당하게 됩니다.

## 21. Model Evaluation

모델 선택은 감으로 하지 않고 리포트로 검증했습니다.

속도, 비용, 정확도, 그리고 MBTI 말투 재현을 비교했고, 이 결과를 서버 라우팅 기준으로 삼았습니다.

핵심은 "어떤 모델이 제일 유명한가"가 아니라, Buddy의 실제 사용 흐름에서 어떤 모델이 빠르고 안정적인가였습니다.

## 22. Eval Reports

검증 리포트의 핵심 결론만 짚겠습니다.

빠른 기본 응답은 GPT-5.4 mini가 유리했고, 감정형 말투는 Claude Haiku가 더 자연스러웠습니다.

또 긴 스크린샷 편집은 텍스트 보존 리스크가 있어, 서버는 이해에 집중하고 Unity가 편집을 담당하게 했습니다.

## 23. API Decisions

이 검증 결과를 API 설계에 반영했습니다.

Unity는 단일 JSON API를 호출하고, 모델 라우팅과 API Key 관리는 서버가 맡는 구조로 정리했습니다.

그래서 클라이언트는 단순하게 유지하고, 품질과 비용, 지연시간 개선은 서버에서 계속 조정할 수 있게 했습니다.

## 24. AI Token Usage

이번 프로젝트는 AI 토큰 지원을 받은 만큼, 단순 코드 생성이 아니라 검증과 제작 속도를 높이는 데 집중적으로 사용했습니다.

현우는 모델 테스트, 정호는 MBTI 말투 검증, 영민은 Runtime UI 표준화, 해민과 승현은 Unity 개발과 Photon 연동 보조에 활용했습니다.

즉, AI를 대신 만들어주는 도구가 아니라 팀의 반복 작업을 줄여주는 협업 도구로 썼습니다.

## 25. PersonaForge Judge

PersonaForge는 AI를 평가자로 쓴 사례입니다.

성격 말투를 생성하고, 이름표를 제거한 뒤, 다른 AI가 어떤 성격처럼 보이는지 판단하게 했습니다.

이 과정을 반복하면서 MBTI 말투 품질을 개선했고, 성격 질문 적중률과 비서형 응답 품질을 함께 끌어올렸습니다.

## 26. Runtime UI Standardization

Runtime UI 표준화는 AI를 반복 개선 파트너로 쓴 사례입니다.

처음에는 Canvas마다 버튼, 색상, 닫기 위치가 달라 UI 톤이 흔들렸습니다.

그래서 먼저 UI 명세를 만들고, 스크린샷 피드백을 좌표와 규칙으로 바꾸며 Canvas별 UI 톤을 맞췄습니다.

## 27. Impact

Buddy가 줄이고 싶은 것은 단순한 기능 클릭 수가 아니라 작업의 마찰입니다.

Before에서는 사용자가 화면 맥락을 직접 캡처하고, 업로드하고, 다시 설명해야 했습니다.

After에서는 선택한 화면이 바로 AI 입력이 되고, 요약, 번역, 설명, 자유 질문이 현재 화면에서 이어집니다.

더 나아가 멀티 클립보드와 상태 공유를 통해 동료와 함께 일하는 맥락까지 이어주는 방향으로 확장할 수 있습니다.

## 28. Roadmap

발표 이후에는 MVP를 제품 경험으로 확장하고 싶습니다.

지금은 캡처, AI 응답, 캐릭터 말풍선 플로우를 안정적으로 보여주는 것이 목표입니다.

다음 단계에서는 스크롤 캡처, 클립보드 히스토리, 최근 기능 모아보기, 단축키 지원처럼 사용성을 강화할 수 있습니다.

이후에는 멀티 클립보드, 팀 상태 공유, 캐릭터 동기화, 성장형 커스터마이징으로 팀 협업 도구까지 확장할 수 있습니다.

## 29. Thanks

이상으로 Buddy 팀 발표를 마치겠습니다.

저희는 AI 기능을 새로 발명하기보다, 사용자가 이미 보고 있는 화면과 AI 사이의 거리를 줄이고, 함께 일하는 동료와의 맥락까지 이어주는 데 집중했습니다.

문제 정의는 초기 회의에서 캡처, AI 서비스 이동, 프롬프트 작성의 뎁스를 줄이는 방향으로 수렴했다는 점입니다.

기술 선택은 Unity, Mac mini 서버, Photon의 역할을 분리했다는 점입니다.

품질 검증은 모델 평가, MBTI 말투, Runtime UI 명세와 audit로 결정했다는 점입니다.

확장 질문은 스크롤 캡처, 팀 클립보드, 성장/미션, 캐릭터 동기화를 후속 확장으로 남겼다고 답하면 됩니다.

## 31. Q&A Unity Client Flow

Unity 안의 처리 흐름은 세 단계입니다.

첫째, 사용자가 전체 화면 또는 필요한 영역을 고릅니다. 둘째, Unity가 화면을 PNG 이미지로 준비합니다.

셋째, 목적에 따라 로컬 처리, 서버 분석, Unity 편집으로 분기합니다.

OCR과 번역처럼 빠르게 처리할 수 있는 것은 로컬에서, 이미지 이해와 persona 응답은 서버의 `/v2/analyze`로 보냅니다.

## 32. Q&A API Surface

Unity는 복잡한 API를 많이 알 필요 없이 몇 개의 명확한 API만 호출하면 됩니다.

핵심은 `POST /v2/analyze`입니다. 캡처 이미지, OCR 텍스트, persona, mode를 보내면 번역, 설명, 질문 응답을 JSON으로 받습니다.

그 외에 저장된 결과 조회, persona 목록, MBTI 퀴즈, 후속 질문 대화 타임라인 API를 준비했습니다.

MVP에서는 Unity가 다루기 쉬운 단일 JSON 요청/응답을 중심으로 안정성을 높였습니다.

## 33. Q&A MBTI Flow

MBTI는 Buddy의 응답 말투를 조정하는 데 사용했습니다.

사용자는 16개 MBTI 중 하나를 직접 고르거나, 퀴즈 결과를 사용할 수 있습니다.

서버는 선택된 유형에 맞춰 답변 톤을 조정하고, 정확성이 더 중요한 상황에서는 기본 말투로 전환할 수 있게 했습니다.

즉, MBTI는 답변의 성격을 만드는 장치이지만, 사실성과 정확성보다 앞서지는 않습니다.

## 34. Q&A MVP Risk

실패하거나 어려웠던 점을 물으면, 아이디어가 넓어질수록 MVP가 흐려졌다고 답하면 됩니다.

처음에는 캐릭터, 캡처, 클립보드, 멀티, 성장 요소까지 모두 가능해 보였지만, 발표 가능한 핵심은 캡처 기반 AI 플로우였습니다.

그래서 캡처, AI 응답, 현재 화면에서 확인하는 흐름으로 다시 좁혔습니다.

## 35. Q&A Character

왜 캐릭터가 필요한지 물으면, 기능만 제공하는 도구보다 현재 화면에 머무르는 Buddy가 더 낮은 진입 장벽과 친근한 상호작용을 만든다고 답하면 됩니다.

캐릭터는 단순 장식이 아니라, 사용자가 AI를 부르는 접점이자 결과를 현재 화면에서 받아들이는 인터페이스입니다.

## 36. Q&A Difference

기존 캡처/AI 도구와의 차이는 캡처와 AI 응답을 한 화면 안에서 이어 붙였다는 점입니다.

핵심은 기능 수를 늘리는 것이 아니라, "캡처 → 이동 → 설명" 흐름을 현재 화면 안에서 줄이는 것입니다.

페르소나와 캐릭터는 반복 사용 경험을 강화하는 장치입니다.

## 37. Q&A Privacy

개인정보와 API Key를 물으면, Claude/OpenAI Key는 Unity가 아니라 Mac mini 서버 환경변수로 관리한다고 답하면 됩니다.

사용자가 명시적으로 캡처한 화면만 서버에 전송하고, 클라이언트에는 모델 Key를 노출하지 않습니다.

서버에서 요청과 응답을 통제하는 구조라 이후 로그 관리나 모델 라우팅도 한 곳에서 할 수 있습니다.

## 38. Q&A Unity

왜 Unity를 검토했는지 물으면, 데스크톱 캐릭터와 비주얼, 에셋 활용에 강점이 있었기 때문이라고 답하면 됩니다.

화면 위에 머무르는 Buddy, 호버 메뉴, 캐릭터 커스터마이징, 캡처 UI를 짧은 시간 안에 확인하기에 적합했습니다.

## 39. Q&A Persona Quality

MBTI 페르소나가 답변 품질을 해치면 어떻게 하는지 물으면, 페르소나는 말투와 우선순위에만 반영한다고 답하면 됩니다.

사실성, 정확성은 기본 응답 품질을 먼저 통과해야 하고, 품질이 흔들리면 기본 모드로 전환할 수 있게 설계했습니다.

## 40. Q&A Why MBTI

왜 페르소나를 MBTI로 잡았는지 물으면, 짧은 해커톤 안에서 설명 가능한 성향 축이 필요했다고 답하면 됩니다.

MBTI는 16개 유형과 말투 차이를 빠르게 만들 수 있고, 사용자가 직접 선택하거나 퀴즈로 진입하기 쉽습니다.

그래서 초기 페르소나 실험에 적합했습니다.

## 41. Q&A Local OCR

OCR/번역을 왜 AI 서버와 분리했는지 물으면, 비용과 지연시간을 줄이기 위해서라고 답하면 됩니다.

번역처럼 반복적이고 즉시성이 중요한 작업은 Tesseract와 무료 번역 경로로 처리하고, AI 서버는 이미지 이해와 질문 응답처럼 판단이 필요한 작업에 집중시켰습니다.

## 42. Q&A Mac Mini

왜 Mac mini 서버를 따로 두었는지 물으면, 보안과 통제를 위해서라고 답하면 됩니다.

Unity 클라이언트에 Claude/OpenAI Key를 넣지 않고, 모델 라우팅과 로그 저장을 서버 한 곳에서 관리했습니다.

클라이언트는 단일 JSON API만 호출하므로 시연과 디버깅도 단순해집니다.

## 43. Q&A Image Edit

이미지 편집도 AI가 직접 하면 안 되는지 물으면, 긴 스크린샷이나 텍스트가 많은 이미지에서는 AI 편집이 글자를 깨뜨릴 수 있다고 답하면 됩니다.

그래서 서버는 화면 이해와 답변 생성에 집중하고, 펜, 텍스트 주석, 저장, 복사는 Unity 클라이언트에서 직접 처리하도록 나눴습니다.

## 44. Q&A Photon

멀티 기능이 MVP에 꼭 필요한지 물으면, MVP의 핵심은 캡처 기반 AI라고 답하면 됩니다.

다만 Photon 멀티, 이모지, 외형 동기화는 Buddy가 개인 도구에서 팀 경험으로 확장될 수 있음을 보여주는 백업 근거로 준비했습니다.

## 45. Q&A AI Usage

AI를 개발 과정에서 어떻게 활용했는지 물으면, 코드 작성만 맡긴 것이 아니라고 답하면 됩니다.

모델 평가, MBTI 말투 검증, Unity Runtime UI 명세화, 스크린샷 기반 레이아웃 반복 개선에 사용했습니다.

특히 UI는 명세, 구현, 검증, audit 흐름으로 통제했습니다.

## 46. Q&A Demo Fallback

시연이 불안정하면 어떻게 설명할 수 있는지 물으면, 핵심 흐름을 분해해서 설명하면 됩니다.

영역 선택, 캡처 이미지 준비, 로컬 OCR/번역 또는 서버 분석, Unity 결과 표시가 핵심입니다.

데모 치트와 저장된 결과 화면은 발표 중 같은 상태를 빠르게 재현하기 위한 장치입니다.

## 47. Q&A Next Step

앞으로 가장 먼저 개선할 부분은 시연 안정성, OCR/이미지 이해 품질, 캡처 후 액션 선택 UX입니다.

그 다음에는 스크롤 캡처, 클립보드 히스토리, 단축키, 멀티 클립보드, 팀 상태 공유로 확장할 수 있습니다.
