# Buddy 발표 대본 초안

기준: 10분 발표. 슬라이드가 많으므로 한 장당 10~25초로 짧게 넘기고, 데모 영상 구간에서 60~90초를 확보하는 흐름입니다.

## 전체 흐름 한 줄

Buddy는 "화면을 캡처하면 AI가 바로 이해하고, 데스크톱 캐릭터가 말풍선으로 도와주는 AI 작업 보조 도구"입니다.

## 1. Title

안녕하세요. 저희는 MODAC-THON 2026 Buddy 팀입니다.

저희가 만든 Buddy는 화면 옆에서 사용자의 맥락을 같이 보고, 필요한 순간에 바로 도와주는 데스크톱 AI 친구입니다.

핵심 키워드는 세 가지입니다. 화면 캡처, AI Assistant, 그리고 Persona Buddy입니다.

## 2. Agenda

오늘 발표는 먼저 팀과 문제 정의를 소개하고, 그 문제를 Buddy라는 MVP로 어떻게 좁혔는지 말씀드리겠습니다.

이어서 핵심 기능, Unity 클라이언트, 서버 구조, AI 모델 검증, 그리고 팀원들이 AI를 어떻게 활용했는지까지 보여드리겠습니다.

마지막에는 시연 영상과 향후 확장 방향, 예상 질문까지 짧게 정리하겠습니다.

## 3. Team Photo

안녕하세요, Buddy 팀입니다.

저희는 이번 모닥톤에서 처음 모였지만, 프론트엔드, Unity, 서버, UI/UX, 멀티 연동처럼 서로 다른 강점을 가진 팀원들이 모였습니다.

그래서 짧은 시간 안에 기획, 구현, 검증, 발표 준비를 병렬로 나눠서 진행할 수 있었습니다.

## 4. Team Members

역할은 각자의 강점에 맞춰 나눴습니다.

현우는 문서 관리와 발표 구조, AI 모델 테스트와 MBTI 데이터 수집을 맡았습니다.

해민, 승현, 영민은 Unity 클라이언트, 에셋 적용, UI 개선, Photon 기반 멀티 연동을 중심으로 작업했습니다.

정호는 MBTI 서버와 API 테스트, 클립보드 기능, AI 서버 구조를 맡았습니다.

## 5. Team Role Breakdown

이 장표는 각자의 작업이 어떻게 연결됐는지를 보여줍니다.

중요했던 점은 단순히 기능을 나눠서 만드는 것이 아니라, 캡처 기반 AI라는 핵심 플로우를 중심에 두고 클라이언트, 서버, AI 검증, UI 작업을 맞춰갔다는 점입니다.

## 6. Problem

저희가 출발한 문제는 AI 자체가 아니라, AI를 사용하는 과정의 깊이였습니다.

실제로 화면에서 궁금한 것이 생기면 스크린샷을 찍고, AI 서비스로 이동하고, 이미지를 올리고, 다시 프롬프트로 맥락을 설명해야 합니다.

이 과정에서 작업 흐름이 끊기고, 같은 맥락을 반복해서 설명하게 됩니다.

그래서 저희는 "화면을 보고 있는 그 자리에서 바로 AI 도움을 받을 수 없을까?"라는 질문에서 시작했습니다.

## 7. Catchphrase

Buddy의 프로젝트 메시지는 "일하는 우리의 삶을 이어지게 만드는 작은 실험"입니다.

여기서 이어진다는 말은 앱과 앱 사이, 화면과 AI 사이, 그리고 사용자와 도구 사이의 흐름이 끊기지 않는다는 뜻입니다.

## 8. Buddy Introduction

Buddy는 같은 곳을 바라보며, 맥락과 질문을 이어주는 동료입니다.

사용자가 보고 있는 화면을 함께 입력으로 보고, 사용자의 성향에 맞춰 필요한 답을 이어주는 데스크톱 AI 동료를 목표로 했습니다.

단순한 챗봇보다, 화면 위에 머무르는 작은 동료에 가깝게 만들고 싶었습니다.

## 9. Insight

해커톤 안에서 모든 것을 만들 수는 없기 때문에 방향을 세 가지로 좁혔습니다.

첫째, 핵심 기능은 캡처한 화면을 AI가 이해하고 바로 응답하는 흐름으로 잡았습니다.

둘째, 보는 즉시 제품 콘셉트가 이해되도록 비주얼 완성도를 확보했습니다.

셋째, MBTI 페르소나와 커스터마이징을 통해 단순 도구보다 동료처럼 느껴지도록 했습니다.

## 10. Concept

Buddy는 화면 위에 상주하는 AI 작업 보조 도구입니다.

평소에는 화면 한쪽에 머무르다가, 사용자가 필요한 순간 화면을 캡처하면 AI가 내용을 이해하고 말풍선으로 답합니다.

한 줄로 말하면 Capture, Understand, Reply입니다.

## 11. User Flow

사용 흐름은 간단합니다.

먼저 프로그램을 켜고 닉네임과 서버 연결을 확인합니다. 그 다음 MBTI를 직접 고르거나 퀴즈를 통해 페르소나를 정합니다.

이후 Buddy가 화면에 등장하고, 사용자가 전체 화면이나 특정 영역을 캡처하면 번역, 설명, 질문 같은 AI 액션을 선택할 수 있습니다.

결과는 현재 화면 안에서 말풍선이나 결과 창으로 바로 확인합니다.

## 12. Feature Capture

첫 번째 핵심 기능은 캡처 기반 AI입니다.

사용자가 화면 전체나 영역을 선택하면, 그 화면이 AI의 입력으로 바뀝니다.

이를 통해 텍스트 추출, 번역, 요약, 이미지 설명, 자유 질문을 현재 화면 안에서 이어갈 수 있습니다.

목표는 반복되는 "캡처, 이동, 설명" 과정을 줄이는 것입니다.

## 13. Feature Persona

두 번째 핵심 기능은 페르소나 버디입니다.

같은 답변이라도 어떤 사용자는 핵심을 빠르게 정리해주길 원하고, 어떤 사용자는 맥락을 부드럽게 설명해주길 원합니다.

그래서 MBTI 기반으로 빠른 정리형, 맥락 공감형, 탐구 질문형처럼 말투와 우선순위를 다르게 가져가도록 설계했습니다.

다만 페르소나는 사실을 바꾸는 것이 아니라, 답변의 톤과 표현 방식을 조정하는 역할입니다.

## 14. Feature Multi

확장 기능으로는 멀티와 클립보드를 검토했습니다.

개인 도구로 시작하지만, 팀원이 복사한 자료를 공유하거나 현재 상태를 이모지로 표현하거나, 서로의 Buddy를 화면에 같이 표시하는 방향입니다.

이번 MVP에서는 핵심 플로우를 우선했고, 멀티 기능은 팀 도구로 확장할 수 있는 가능성으로 열어뒀습니다.

## 15. Demo

이제 시연 영상에서는 병목이 실제로 줄어드는 순간을 보여드리겠습니다.

먼저 온보딩에서 닉네임과 MBTI 페르소나를 선택합니다.

그 다음 화면 일부를 캡처하고, 번역이나 설명 같은 액션을 실행합니다.

마지막으로 AI 응답이 현재 화면 안에서 결과 창이나 말풍선으로 이어지는 흐름을 보시면 됩니다.

데모에서 봐주실 포인트는 기능 개수보다, "AI를 쓰기 위해 화면 밖으로 나가지 않아도 된다"는 경험입니다.

## 16. Server Architecture

전체 구조는 Unity 데스크톱 클라이언트와 Mac mini FastAPI 서버로 나눴습니다.

Unity는 화면 캡처, 캐릭터, UI, 사용자 입력을 담당하고, 서버는 AI 모델 호출, persona 적용, 결과 저장을 담당합니다.

Claude와 OpenAI API 키는 클라이언트가 아니라 서버 쪽에서 관리해서 보안과 라우팅을 한 곳에 모았습니다.

## 17. Technical Approach

기술 접근은 짧은 시간 안에 시연 가능한 조합을 기준으로 잡았습니다.

클라이언트는 Unity와 C#을 중심으로 데스크톱 앱, 캐릭터, 캡처, 에셋 활용을 검토했습니다.

AI 서버는 같은 Wi-Fi의 Mac mini에서 FastAPI로 실행했고, 모델 라우팅과 결과 저장을 맡겼습니다.

Persona는 MBTI 질문셋과 말투 프롬프트로 구성했고, Multi는 Photon으로 접속과 동기화를 검토했습니다.

## 18. Unity Client Experience

실제 사용 경험은 Unity가 만들었습니다.

투명 오버레이 창 위에 2D 캐릭터가 떠 있고, 필요한 순간 호버 메뉴로 기능을 열 수 있습니다.

캡처, OCR/번역, AI 분석 결과, 말풍선 표시까지 Unity 안에서 이어지도록 구성했습니다.

또한 커스터마이징, MBTI 말투, 이모지 표현, Photon 기반 멀티 표현도 Unity 클라이언트가 담당했습니다.

## 19. Unity Client Flow

Unity 안의 처리 흐름은 세 단계입니다.

첫째, 사용자가 전체 화면 또는 필요한 영역을 고릅니다. 둘째, Unity가 화면을 PNG 이미지로 준비합니다.

셋째, 목적에 따라 로컬 처리, 서버 분석, Unity 편집으로 분기합니다.

OCR과 번역처럼 빠르게 처리할 수 있는 것은 로컬에서, 이미지 이해와 persona 응답은 서버의 `/v2/analyze`로 보냅니다.

## 20. API Surface

Unity는 복잡한 API를 많이 알 필요 없이 몇 개의 명확한 API만 호출하면 됩니다.

핵심은 `POST /v2/analyze`입니다. 캡처 이미지, OCR 텍스트, persona, mode를 보내면 번역, 설명, 질문 응답을 JSON으로 받습니다.

그 외에 저장된 결과 조회, persona 목록, MBTI 퀴즈, 후속 질문 대화 타임라인 API를 준비했습니다.

MVP에서는 Unity가 다루기 쉬운 단일 JSON 요청/응답을 중심으로 안정성을 높였습니다.

## 21. MBTI Flow

MBTI는 Buddy의 응답 말투를 조정하는 데 사용했습니다.

사용자는 16개 MBTI 중 하나를 직접 고르거나, 퀴즈 결과를 사용할 수 있습니다.

서버는 선택된 유형에 맞춰 답변 톤을 조정하고, 정확성이 더 중요한 상황에서는 기본 말투로 전환할 수 있게 했습니다.

즉, MBTI는 답변의 성격을 만드는 장치이지만, 사실성과 정확성보다 앞서지는 않습니다.

## 22. Model Evaluation

모델 선택은 감으로 하지 않고 리포트로 검증했습니다.

번역, 질문, 설명에서 모델별 품질, 비용, 지연시간을 비교했고, 빠르고 비용 효율적인 기본 응답은 GPT-5.4 mini 중심으로 판단했습니다.

반면 감정형 말투처럼 부드러운 표현이 중요한 영역에서는 Claude Haiku가 더 자연스러운 결과를 보였습니다.

그래서 서버가 상황에 따라 모델을 라우팅할 수 있는 구조를 선택했습니다.

## 23. Eval Reports

검증 리포트의 결론은 세 가지입니다.

첫째, 작업 응답 속도 기준으로 GPT-5.4 mini가 빠르고 정확도도 충분했습니다.

둘째, MBTI 말투 재현에서는 ENFP나 ENFJ 같은 감정형에서 Claude Haiku가 더 자연스러웠습니다.

셋째, 긴 스크린샷을 AI가 직접 편집하는 방식은 텍스트 보존 리스크가 있어, 서버는 이해에 집중하고 실제 편집은 Unity가 담당하는 쪽으로 정리했습니다.

## 24. API Decisions

실험 결과는 서버 API 설계에도 반영했습니다.

처음에는 병렬 요청도 검토했지만, 시연 안정성을 위해 `/v2/analyze` 단일 요청 중심으로 정리했습니다.

Unity는 persona와 mode만 전달하고, 모델 라우팅과 API Key 관리는 서버가 맡습니다.

또한 응답에 모델명과 처리 시간을 저장해서 이후 품질, 비용, 지연시간 개선의 근거로 사용할 수 있게 했습니다.

## 25. AI Token Usage

이번 프로젝트에서 AI 토큰은 단순 생성보다 검증과 제작 속도를 높이는 데 사용했습니다.

현우는 모델 테스트와 비용, 속도, 정확도 비교를 맡았고, 정호는 MBTI 페르소나와 PersonaForge 검증 구조를 만들었습니다.

영민은 Runtime UI 표준화에 Claude Code를 사용했고, 해민과 승현은 Unity 개발과 Photon 연동 과정에서 AI를 개발 보조로 활용했습니다.

중요했던 점은 AI를 "대신 만들어주는 도구"가 아니라, 검증과 반복을 빠르게 해주는 협업 도구로 사용했다는 점입니다.

## 26. PersonaForge Judge

정호는 MBTI 말투를 AI 심판으로 반복 검증했습니다.

먼저 특정 성격처럼 답변을 생성하고, 이름표를 제거한 뒤 다른 AI가 어떤 성격처럼 보이는지 판단하게 했습니다.

이 과정을 반복하면서 유형별, 계열별, E/I, S/N, T/F, J/P 축별로 어디가 잘 맞고 어디가 흔들리는지 확인했습니다.

그 결과 성격 질문 적중률을 높이고, 비서형 응답 품질도 개선할 수 있었습니다.

## 27. Runtime UI Standardization

영민은 AI를 UI 표준화 파이프라인으로 사용했습니다.

처음에는 Canvas마다 버튼, 색상, 닫기 위치, Text/TMP 사용 방식이 달라 UI 톤이 흔들렸습니다.

그래서 AI에게 바로 코드를 맡기지 않고, 먼저 UI 명세를 만들고 그 명세를 기준으로 Canvas 구현을 맞췄습니다.

이후 스크린샷 피드백을 좌표, 색상, 규칙으로 바꾸고, git diff 검증과 audit 문서로 반복했습니다.

결과적으로 Panel, Button, Close, Animation, Window exclusivity 같은 공통 규칙을 정리할 수 있었습니다.

## 28. Impact

Buddy가 줄이고 싶은 것은 단순한 기능 클릭 수가 아니라 작업의 마찰입니다.

Before에서는 사용자가 화면 맥락을 직접 캡처하고, 업로드하고, 다시 설명해야 했습니다.

After에서는 사용자가 선택한 화면이 바로 AI 입력이 되고, 요약, 번역, 설명을 현재 화면에서 확인할 수 있습니다.

이 흐름은 개발자뿐 아니라 기획자, 디자이너, 일반 사용자에게도 적용될 수 있습니다.

## 29. Roadmap

발표 이후에는 MVP를 제품 경험으로 확장하고 싶습니다.

지금은 캡처, AI 응답, 캐릭터 말풍선 플로우를 안정적으로 보여주는 것이 목표입니다.

다음 단계에서는 스크롤 캡처, 클립보드 히스토리, 최근 기능 모아보기, 단축키 지원처럼 사용성을 강화할 수 있습니다.

이후에는 멀티 클립보드, 팀 상태 공유, 캐릭터 동기화, 성장형 커스터마이징으로 팀 협업 도구까지 확장할 수 있습니다.

## 30. Q&A

예상 질문도 몇 가지 준비했습니다.

"왜 캐릭터가 필요한가요?"에 대해서는, 기능만 제공하는 도구보다 현재 화면에 머무르는 Buddy가 더 낮은 진입 장벽과 친근한 상호작용을 만든다고 답할 수 있습니다.

"기존 캡처/AI 도구와 무엇이 다른가요?"에 대해서는, 캡처와 AI 응답을 한 화면 안에서 이어 붙이고, persona와 캐릭터로 반복 사용 경험을 강화한다는 점을 말하면 됩니다.

"개인정보와 API Key는 어떻게 다루나요?"에 대해서는, API Key는 서버 환경변수로 관리하고, 사용자가 명시적으로 캡처한 화면만 전송한다고 답하면 됩니다.

## 31. Q&A Persona

페르소나 관련 리스크도 준비했습니다.

MBTI가 답변 품질을 해칠 수 있다는 질문에는, 페르소나는 말투와 우선순위에만 반영하고 사실성과 정확성은 기본 응답 품질을 먼저 통과한 뒤 적용한다고 답하면 됩니다.

어려웠던 점은 아이디어가 넓어질수록 MVP가 흐려졌고, 캡처 기반 AI라는 핵심 플로우로 다시 좁히는 과정이었다고 말할 수 있습니다.

앞으로는 시연 안정성, OCR/이미지 이해 품질, 캡처 후 액션 선택 UX를 우선 개선하려고 합니다.
